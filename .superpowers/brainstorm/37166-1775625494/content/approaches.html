3 approaches to Supabase auth integration

Each has different complexity and trade-offs. My recommendation is A.

A ★

@supabase/ssr — recommended

Supabase's official Next.js App Router package. Handles cookie-based sessions automatically across server components, API routes, and middleware. Replaces iron-session entirely. Clean and well-supported.

Pros

  • Official, maintained by Supabase
  • Works seamlessly in proxy.ts, server components, API routes
  • Handles token refresh automatically

Cons

  • Requires updating proxy.ts and auth routes
B

Supabase Admin SDK (server-only)

Use the Supabase service-role key server-side to verify JWTs. Keep iron-session for the cookie layer. Supabase handles login flow; you keep your existing session infrastructure.

Pros

  • Minimal changes to existing proxy/session code

Cons

  • Two session systems in play (Supabase + iron-session)
  • More complexity, less official support
C

Client-side only (@supabase/supabase-js)

Auth entirely in the browser. No server-side session. Supabase JS client handles login and stores the JWT in localStorage.

Pros

  • Simplest to implement

Cons

  • Can't protect API routes server-side
  • JWT in localStorage is less secure
  • Doesn't work with server components
